## GOAL
Make dedicated MSSQL Windows Jenkins LOAD CDC semantics match the proven local .bat semantics: exit 0 continues, exit 100 skips data load successfully, other non-zero fails the pipeline.

## WHAT_WAS_FOUND
- `scripts/batch/mssql/load/run_cdc.bat` correctly returns exit 100 when no CDC changes are detected.
- Local `scripts/batch/mssql/mssql_load_pipeline.bat` handles exit 100 with `if errorlevel 100 goto skip_data_load`.
- Dedicated Jenkins Groovy `jenkins/mssql/windows/load_pipeline.groovy` invoked `run_cdc.bat` inside `runTrackedStage`, which treats any non-zero exit as Exception → FAILURE → rethrow.
- Result: CDC skip (legitimate no-change case) failed the entire pipeline in Jenkins.

## ROOT_CAUSE
`runTrackedStage` wrapper does not allow inspecting non-zero exit codes from CDC. The proven PostgreSQL Groovy pattern uses `bat(returnStatus: true)` to capture and branch on the exit code.

## CHANGES_MADE

### jenkins/mssql/windows/load_pipeline.groovy
- Replaced `runTrackedStage('Run CDC') { bat '...run_cdc.bat' }` with explicit manual logging using stage-start/end logger calls and `bat(returnStatus: true)`.
- Added branching logic:
  - `cdcResult == 0 || cdcResult == 100` → log SUCCESS, continue.
  - Inside that branch, if `cdcResult == 100` → echo SKIP message, set `env.SKIP_DATA_LOAD = 'true'`.
  - Any other non-zero → log FAILURE, run `set-error`, `error` to fail the stage.
- Added `when { env.SKIP_DATA_LOAD != 'true' }` guards to `Load Data` and `Validate Loaded Data` stages so they are skipped when CDC reports no changes.

No other files modified.
