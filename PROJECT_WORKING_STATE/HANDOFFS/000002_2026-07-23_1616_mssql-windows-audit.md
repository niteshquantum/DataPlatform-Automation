## WHAT_WAS_FOUND

### SETUP Flows
- **Local .bat**: `mssql_setup_pipeline.bat` → validate Python/Java → install tools → check_instance → (NO_INSTANCE) deploy_mssql_gdrive → configure_mssql → start_mssql → validate_environment
- **Dedicated Groovy**: identical logical stages. CRITICAL GAP: `configure_mssql.bat` is gated behind `readFile('admin_status.txt').trim() == 'true'`. .bat runs it unconditionally for NO_INSTANCE. Without admin, Groovy may leave instance unconfigured.

### LOAD Flows
- **Local .bat**: `mssql_load_pipeline.bat` → validate python/runtime → validate py reqs → start_mssql → validate_mssql → create_database → download_dataset → run_cdc (exit-100 = skip) → load_data → validate_loaded_data → deploy_objects → validate_objects. Assessment/reporting commented out.
- **Dedicated Groovy**: `load_pipeline.groovy` → Download Dataset → Create Database → Run CDC (wrapped in runTrackedStage — **BUG**) → Load Data/Skip → Validate → Deploy Objects → Validate Objects → optional Assessment/Report.

### BAT vs Groovy Gaps (Real Parity Gaps)
1. **CRITICAL CDC SKIP BUG**: Groovy invokes `run_cdc.bat` inside `runTrackedStage`. Exit 100 is treated as FAILURE and rethrown. Inline .bat handles it with `if errorlevel 100 goto skip_data_load`. PostgreSQL Groovy uses `returnStatus: true` to avoid this.
2. **CRITICAL FRESH WORKSPACE GAP**: Groovy LOAD does NOT include: validate_python_runtime, validate_python_requirements, install_tools, start_mssql, validate_mssql. .bat LOAD includes all of them. If SETUP and LOAD run in different workspaces, LOAD fails because python deps, tools, and MSSQL instance are absent.
3. **SETUP CONFIGURE GATING**: .bat unconditional vs Groovy admin-gated for configure_mssql.bat on fresh deploy.
4. **Profile Source Data**: Neither .bat nor Groovy calls data_profiler for MSSQL. PostgreSQL Groovy has it; PostgreSQL .bat does not. Cross-database inconsistency, not MSSQL-specific.
5. **Assessment/Reporting wiring**: .bat has assessment commented out; Groovy has optional stages. But underlying MSSQL wrappers are missing.

### Instance Ownership Finding
`scripts/python/mssql/setup/check_instance.py` checks:
- Windows service `MSSQL$DMSQL` exists
- TCP port 1533 listening

It does NOT verify:
- Project-managed binary path
- Project-managed data directory
- Installation provenance
- Instance name matching config beyond service name

**Risk**: A foreign/old SQL Server on port 1533 with a matching service name passes as `INSTANCE_RUNNING_AND_USABLE`. `PORT OPEN != MANAGED INSTANCE OWNERSHIP`.

PostgreSQL `check_instance.py` checks `databases/postgresql/bin/pg_ctl.exe` and `databases/postgresql/data/PG_VERSION` plus `pg_ctl status -D` before declaring ownership. MSSQL has no equivalent.

### Fresh Workspace Risks
- **LOAD in new workspace**: No Python dep validation, no tool install, no MSSQL start/validate. Will fail immediately.
- **Tools/Drivers expectation**: LOAD expects `tools/liquibase/liquibase.bat` and `tools/drivers/mssql-jdbc-12.10.0.jre11.jar` from SETUP workspace.
- **Generated artifacts**: `liquibase/mssql/master.xml` is tracked; numbered changelogs and `master_objects.xml` are gitignored but regenerated by `bootstrap_generator.py` at runtime.
- **Schema Liquibase**: `generate_liquibase_xml.py` and `update_master_xml.py` exist but are never called from any pipeline. `run_liquibase.bat` exists but is never invoked from LOAD or SETUP.

## ROOT_CAUSE
MSSQL Windows implementation was cloned from PostgreSQL Windows baseline but was never fully parity-adjusted. Critical gaps introduced:
1. `check_instance.py` was not adapted for MSSQL ownership verification model
2. Groovy LOAD was not aligned with local .bat or PostgreSQL Groovy fresh-workspace bootstrap
3. CDC exit-100 handling was inherited from PostgreSQL bug pattern (PostgreSQL fixed it, MSSQL was not)
4. Assessment/migration wrappers were not created (no MSSQL-specific wrappers exist)
5. Schema Liquibase evolution scripts were ported but never wired into pipelines
